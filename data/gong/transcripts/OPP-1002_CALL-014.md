# Transcript: CALL-014 - Troubleshooting enablement session

**Opportunity:** OPP-1002  
**Account:** ACC-2002  
**Date:** 2026-04-10  
**Source access level:** standard

Vendor Customer Success: Today's session is a practical walkthrough. We will test whether the troubleshooting flow works for factory operations, not only for the security architecture team.

Clara Esteves, Director of Plant Systems: That is the right frame. The architecture review was useful, but plant users need fewer steps.

Niko Hart, Network Operations Manager: I brought two examples from recent incidents. One was a path failure that recovered quickly. The other was a policy mismatch that looked like a path failure. Operators treated both the same.

Vendor Support Lead: Let's start with the first incident. In the simplified view, the operator sees the site is connected over backup path, last failover occurred four minutes ago, and production apps are reachable.

Niko Hart: That is better. It should say "monitor, do not escalate" if the backup path is healthy. Otherwise the operator still calls central network.

Henrik Vale, Security Architect: Be careful. If the backup path is healthy but the security policy is degraded, we still need escalation.

Vendor Support Lead: We can split the recommendation: connectivity healthy, policy unchanged, no escalation; connectivity healthy, policy degraded, escalate to security.

Clara Esteves: That works if the text is short. Night shift will not read a paragraph.

Vendor Customer Success: For the second incident, the policy mismatch appears as access denied to one application group while the site path is healthy.

Niko Hart: That is exactly where confusion happens. The operator says "network is down" because the app is unavailable.

Henrik Vale: The policy template should name the application group and owner. If it only says denied, it creates a ticket with no owner.

Vendor Support Lead: We can add owner group and policy last changed timestamp.

Clara Esteves: Good. But the runbook still needs to be two layers: quick action for operators, diagnostic detail for supervisors.

Vendor AE: How does this affect proof success?

Julian Maro, CIO: Joining late, but from my perspective, enablement is part of the proof. If the tool works but the team cannot operate it, finance will see that as rollout risk.

Niko Hart: The runbook is close, but not ready. It has too many product terms.

Vendor Customer Success: We will rewrite it in operator language and keep product terms in the appendix.

Henrik Vale: I want the policy drift report included in the same proof pack. Security is aligned, but I need evidence that templates do not drift plant by plant.

Clara Esteves: That is fine, but do not overload the plant proof. We have two audiences.

Vendor AE: So the proof pack should have an executive scorecard, operator runbook, supervisor diagnostics, and security evidence appendix.

Julian Maro: Yes. And keep the scorecard blunt. Green, yellow, red. If reporting export is still failing, call it yellow. Do not bury it.

Niko Hart: I appreciate that. The export failure is not fatal, but pretending it did not happen would damage trust.

Vendor Customer Success: Action items: rewrite runbook, add owner groups, split operator and supervisor steps, add policy drift evidence, and mark reporting export as pending retest.

Clara Esteves: That will help me keep the champion position internally.

Julian Maro: I want to add one executive concern. If the proof pack contains too much technical detail, finance will miss the point. The executive layer should show whether the proof reduces rollout risk.

Vendor AE: We can structure it as scorecard first, then operator appendix, supervisor appendix, and security evidence appendix.

Henrik Vale: That works. Security evidence should not be watered down, but it does not need to be the opening page.

Niko Hart: In the operator appendix, include examples of wrong interpretation. For example, app unavailable does not always mean path down.

Vendor Support Lead: Good idea. We can include a "symptom does not always equal cause" section with two examples.

Clara Esteves: Keep it short. The more elegant the runbook, the more likely plant supervisors will use it.

Julian Maro: What is the open risk after this session?

Niko Hart: Reporting export and ongoing ownership. A runbook can be good on day one and stale on day thirty.

Henrik Vale: Policy drift is another. If factories make local changes without review, the expansion loses its security value.

Vendor Customer Success: Then the close plan should include a thirty-day owner-map review and recurring policy drift review.

Clara Esteves: That is the difference between a proof and an operating model. Put that in the brief.

Vendor AE: Noted. The recommendation will be to proceed with proof closeout only after export retest, owner-map review plan, and policy drift cadence are documented.
